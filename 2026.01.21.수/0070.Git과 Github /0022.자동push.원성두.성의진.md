# Git 자동 푸시(Auto-Push) 설정 가이드

결론부터 말씀드리면, Git 자체에는 '커밋 시 자동 푸시'라는 옵션이 기본적으로 내장되어 있지 않습니다. Git의 철학이 **"로컬에서 완벽하게 검증한 후 서버에 올린다"**이기 때문입니다. 하지만 **Git Hooks**라는 기능을 이용하면 커밋이 성공하는 즉시 자동으로 푸시가 실행되도록 강제할 수 있습니다.

---

### 1. Git Hooks를 이용한 자동화 원리

Git에는 특정 이벤트(커밋, 푸시 등)가 발생할 때 자동으로 실행될 스크립트를 지정할 수 있는 **Hooks** 시스템이 있습니다. 그 중 `post-commit`이라는 훅을 사용하면 커밋 직후에 `git push` 명령어가 실행되도록 만들 수 있습니다.

---

### 2. 자동 푸시 설정 방법 (Step-by-Step)

현재 작업 중인 프로젝트 폴더의 설정 파일만 살짝 수정하면 됩니다.

1. **Hooks 디렉토리로 이동:**
터미널(또는 Git Bash)에서 프로젝트 루트 폴더에 있는 숨겨진 폴더로 들어갑니다.
```bash
cd .git/hooks

```


2. **`post-commit` 스크립트 파일 생성:**
해당 폴더에 `post-commit`이라는 이름의 파일을 만듭니다. (확장자 없음)
3. **스크립트 내용 입력:**
파일을 열고 아래 내용을 작성합니다.
```bash
#!/bin/sh
# 커밋이 완료된 후 현재 브랜치를 origin으로 자동 푸시
git pull origin main
git push origin $(git rev-parse --abbrev-ref HEAD)

```


4. **실행 권한 부여 (Linux/Mac 사용자만 해당):**
파일이 실행될 수 있도록 권한을 설정합니다.
```bash
chmod +x post-commit

```



---

### 3. 결과 확인

이제 평소처럼 파일을 수정하고 커밋만 해보세요.

```bash
git add .
git commit -m "Auto push test"

```

커밋 메시지를 입력하고 엔터를 치는 순간, 터미널에 `Pushing to...`라는 메시지가 뜨면서 별도의 명령 없이도 GitHub/GitLab 서버에 코드가 올라가는 것을 확인할 수 있습니다.

---

### 4. 주의사항 및 권장하지 않는 이유

자동 푸시는 편리해 보이지만, 실무에서는 다음과 같은 이유로 잘 사용하지 않습니다.

* **실수 수정 불가:** 오타가 있거나 잘못된 코드를 커밋했을 때, 로컬에서 수정할 틈도 없이 바로 서버에 공개됩니다.
* **네트워크 의존성:** 커밋할 때마다 인터넷 연결이 확인되어야 하므로, 오프라인 환경에서는 커밋 자체가 느려지거나 오류가 발생할 수 있습니다.
* **커밋 단위 오염:** 서버에는 완성된 기능 단위로 푸시하는 것이 좋은데, 사소한 오타 수정 하나하나가 서버 히스토리에 남게 되어 관리가 힘들어집니다.
